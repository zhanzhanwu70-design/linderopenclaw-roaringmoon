# opencode + Ollama 多 GPU 工作流建議

更新日期：2026-04-02

## 問題背景

硬體候選：
- 2 張 5090 32G
- 2 張 Pro 6000 48G
- RTX4000 SFF 40G

目標：
- 使用 `opencode` 當 CLI coding agent 入口
- 使用 `ollama` 作為模型選擇器 / 本地模型服務層
- 評估適合的工作流、模型分工與部署策略

---

## 先講結論

如果你的目標是 **本地 AI coding agent / CLI 工作流**，我建議的優先順序是：

### 最佳整體方案
1. **主力推理卡：2× Pro 6000 48G**
2. **高性價比 / 高吞吐方案：2× 5090 32G**
3. **低功耗輔助 / 邊角任務：RTX4000 SFF 40G**

### 工作流建議
- **opencode 做前台 CLI**
- **Ollama 做本地模型入口層**
- 採用 **任務分流工作流**，不要期待單一模型包辦全部

### 模型策略建議
- **聊天 / 閱讀 / 輕摘要**：中小型快模
- **coding / repo 理解 / 規劃**：中大型 code / instruct 模型
- **重推理 / 架構分析 / 長上下文**：較大參數模型或雲端補位

---

## 一、硬體評估

## 1. 2× 5090 32G

### 優點
- 通常有非常強的單卡推理性能
- 對 14B / 32B / 部分量化 70B 類模型很有吸引力
- 很適合高吞吐 coding / inference 工作負載

### 風險 / 注意點
- 32G VRAM 雖然很強，但遇到更大的長上下文模型仍會卡住
- 多卡在 Ollama / 本地推理框架下不一定等於「完美線性加速」
- 功耗、散熱、供電與機箱壓力通常很高
- 若工作流高度依賴單大模型常駐，32G 還是可能覺得不夠

### 適合什麼
- 以 **coding agent + 量化模型** 為主
- 重視吞吐 / 速度
- 能接受用不同模型分工，而不是所有事都壓在一個超大模型上

---

## 2. 2× Pro 6000 48G

### 優點
- 48G VRAM 對本地 agent 非常有價值
- 比 32G 更有餘裕承載大模型、較高精度、長上下文
- 對多模型常駐、較大型 code model、較少激進量化更友善
- 若是專業卡，穩定性、驅動與長時間負載表現通常更適合工作站

### 風險 / 注意點
- 成本通常顯著更高
- 若你的主要需求只是中型模型 coding，性能/價格比不一定比 5090 漂亮
- 還是要確認實際框架對多卡切分與排程支援程度

### 適合什麼
- 想把本地 agent 環境做成 **長期工作站**
- 想讓更大模型在本地可用
- 想保留更多未來升級空間

---

## 3. RTX4000 SFF 40G

### 優點
- 40G 很實用，不是玩具卡
- SFF / 功耗 / 空間通常更容易部署
- 適合當輔助推理卡、embedding / rerank / 背景服務卡

### 風險 / 注意點
- 如果你的目標是「主力 coding 大模型」，它未必是最強主卡
- 在極重負載與高吞吐下，通常不如前兩組雙卡方案
- 若要做主力模型服務，可能會在速度上輸給 5090 級別

### 適合什麼
- 當 **副角色 GPU**
- 跑 embedding、reranker、輕量 coding 模型、常駐小模型
- 做 always-on service 很合適

---

## 二、最推薦的角色分工

## 方案 A：雙層工作流（最推薦）

### GPU 分工
- **主推理層**：2× Pro 6000 48G 或 2× 5090 32G
- **輔助服務層**：RTX4000 SFF 40G

### 任務分工
- 主卡：大語言模型（coding / planning / repo reasoning）
- RTX4000 SFF：
  - embeddings
  - reranker
  - 輕量摘要模型
  - 背景文檔整理 / 監控 / 輔助 agent

### 這樣做的好處
- 不讓大模型被「小事」塞住
- coding agent 的主模型可穩定常駐
- 背景索引、檢索、摘要不會搶主模型資源

---

## 方案 B：主力單機 coding box

如果你只想做一台主力機，不想搞太多分層：

### 優先順序
1. **2× Pro 6000 48G**：最完整、最像工作站
2. **2× 5090 32G**：追求速度與性價比

### 適合工作流
- `opencode` 直接接本地 Ollama
- 一個主模型處理 coding
- 一個輕模型處理讀檔、摘要、簡單問答

---

## 三、opencode 當 CLI 時，工作流怎麼選

## 核心原則
不要用「單模型處理所有任務」思維。
應該用：

### 1. 快模處理輕任務
- 閱讀
- 摘要
- 閒聊
- 小型 repo 掃描

### 2. 強模處理重任務
- coding
- debug
- 架構設計
- refactor
- 長上下文理解

### 3. 必要時雲端補位
有些任務在本地仍然不划算，例如：
- 特別長上下文
- 特別難的架構推理
- 成本不敏感但求成功率最高

這種情況可以保留一條雲端 fallback。

---

## 我建議的 CLI 工作流

### Workflow 1：日常開發
- `opencode` 為唯一入口
- 預設先走本地快模
- 遇到 coding / edit / debug / plan 時切到主力 code model
- 遇到超重任務時，允許手動切更大本地模型或雲端模型

### Workflow 2：研究 / 閱讀型
- 先用輕模型做 repo / docs 初掃
- 再把整理後的 context 交給強模型做判斷
- 好處是省 VRAM、也省等待時間

### Workflow 3：多 agent 分工
如果 `opencode` 支援多代理或你願意自己包一層：
- reader agent：讀檔 / 摘要
- coder agent：寫 code / debug
- planner agent：做規劃 / 架構判斷

這種工作流比「單 agent 強迫切模型」更穩。

---

## 四、Ollama 當模型選擇器，模型怎麼配

先講原則：

### 不要只選一個模型
建議至少準備三層：
1. **快模**
2. **主力 coding 模型**
3. **重推理 / 大任務模型**

---

## 建議模型分層

## A. 快模（聊天 / 閱讀 / 摘要）
目標：低延遲、便宜、可常駐

可考慮類型：
- 7B–14B instruct 類模型
- 較快的 Qwen / Llama / Gemma 類 instruct 變體

用途：
- 閒聊
- docs 閱讀
- PRD / log / 網頁摘要
- 第一輪檔案理解

### 建議原則
- 重視延遲而不是極限智商
- 優先選中文/英文都穩的 instruct 模型

---

## B. 主力 coding 模型
目標：日常寫 code、debug、repo 操作

可考慮類型：
- 14B–32B code/instruct 模型
- 擅長工具使用、repo 理解、patch 產生的模型
- 優先考慮 Qwen code 系、DeepSeek code/instruct 系、Llama code 能力較佳變體

用途：
- 寫函式
- debug
- repo 閱讀後提出 patch
- shell / build 錯誤修正
- 產生 commit-ready 變更

### 建議原則
- 若是本地 coding agent，**主力模型最重要**
- 先把這層調到最好，再考慮更大的研究模型

---

## C. 重推理 / 架構模型
目標：比較方案、做長篇研究、架構判斷

可考慮類型：
- 32B–70B 級量化模型
- 長上下文能力較好的 instruct 模型

用途：
- 系統架構比較
- 跨 repo 分析
- 長文總結
- 高難度規劃

### 注意
- 本地 70B 即使可跑，也不一定代表互動體驗好
- 若等待時間太長，反而破壞 CLI workflow
- 所以這層可以是「按需啟用」，不要常駐

---

## 五、對你這組硬體的實際建議

## 推薦策略 1：以 Pro 6000 為主
如果你真的要打造 **本地 AI coding 工作站**，我最偏好：

- **主力**：2× Pro 6000 48G
- **輔助**：RTX4000 SFF 40G

### 原因
- 48G VRAM 很適合本地 coding / planning / 長上下文
- 更容易容納較大模型與較少量化壓縮
- RTX4000 SFF 可以分擔 embedding / 小模型 / 背景工作
- 這會是一套比較「像平台」而不是「像玩具」的配置

---

## 推薦策略 2：以 5090 為主
如果你重視：
- 速度
- 性價比
- 日常 coding 吞吐

則可選：
- **主力**：2× 5090 32G
- **輔助**：RTX4000 SFF 40G

### 原因
- 對日常 coding 其實很夠用
- 中型模型會很舒服
- 比較適合高互動頻率的 CLI 使用感

### 代價
- 面對更大模型與更高 VRAM 任務時，不如 48G 安心

## 六、落地建議（重要）

## 1. 不要一開始就追求單大模型全包
先建立：
- 快模
- coding 主模
- 大推理模

三層即可。

## 2. 讓 Ollama 只做 serving，不要讓它承擔所有 orchestration 複雜度
Ollama 很適合當本地模型入口，
但真正的任務分流、agent routing、workflow policy，最好交給上層工具或你自己的規則。

## 3. 先優化 coding 主模型，不要先沉迷最大模型
對 `opencode` 這種 CLI agent 來說，
**主力 coding 模型的穩定性與延遲** 比 70B 是否能跑更重要。

## 4. 保留雲端 fallback
即使你是本地派，也建議保留：
- GPT / Claude / Gemini 類雲端 fallback

理由很簡單：
- 少數高難度任務，雲端模型仍然更划算

## 5. 若要長期維護，建立「模型路由表」
例如：
- 閱讀 / 摘要 → 快模
- coding / debug → 主力 code model
- 架構 / 研究 → 大模型
- 超重任務 → 雲端 fallback

這比「看心情切模型」可靠很多。

## 七、我目前的具體推薦

如果是我幫你選：

### 偏工作站穩定派
- **首選硬體**：2× Pro 6000 48G + RTX4000 SFF 40G
- **工作流**：opencode + Ollama + 三層模型路由
- **模型策略**：
  - 快模：7B–14B instruct
  - 主力 coding：14B–32B code/instruct
  - 研究模型：32B–70B 按需啟用

### 偏高互動性價比派
- **首選硬體**：2× 5090 32G + RTX4000 SFF 40G
- **工作流**：同上
- **模型策略**：相同，但更偏向把主力放在 14B–32B，少碰超大模型常駐
